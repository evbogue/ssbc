# Work Order: Serve our own docs at `/docs` with a generated, complete built-in API reference

**Status:** Ready
**Depends on:** none. Supersedes task 3a ("archive framing note") of `docs-and-readme-work-order.md`: instead of caveating the archived site, we replace it as the primary `/docs` surface and relabel the archive. The four doc-to-code drift fixes from that order are already landed.
**Intent:** Make `http://127.0.0.1:8989/docs` serve *this repository's* Markdown docs (`docs/*.md`) as rendered HTML, with a **generated** API reference that lists the complete built-in static RPC surface so coverage cannot silently fall behind the code. Demote the vendored scuttlebot.io site to a clearly-labelled historical archive at `/docs/archive`.

## Why this work order exists

`/docs` currently serves `docs/scuttlebot.io/` — a built copy of the original scuttlebot.io website (~2016), vendored in `vendor/scuttlebot.io/` and regenerated by `npm run sync:scuttlebot-docs`. It documents a different, older project (flume storage, pre-EBT replication, Node-only clients) and is structurally blind to what `ssbc` actually is (SQLite store, EBT, ssb-ws, git-over-HTTP, Decent/ssbski). Re-running sync reproduces the same stale content; it is an archive, not our documentation.

Meanwhile we already maintain accurate Markdown docs in `docs/` that nothing surfaces at `/docs`. And `docs/api.md` is an *overview*, not a reference: of the 26 core manifest methods it names 17 and misses 9 (`del`, `queue`, `flush`, `search`, `createWriteStream`, `getVectorClock`, `getAtSequence`, `ready`, `version`), and it documents almost none of the 57 built-in plugin manifest methods (`gossip.*`, `blobs.want`, `invite.create`, and `ebt.*`, among others).

The fix has two halves:
1. **Repoint** `/docs` at `docs/*.md`, rendered to HTML, archive moved to `/docs/archive`.
2. **Generate** a complete API reference from the built-in manifests so the repository's static RPC surface is always listed, with hand-written prose attached incrementally.

Before either half ships, audit the canonical pages for current truth. Publishing a page at
`/docs` makes it primary documentation; proposals and work orders must not appear there as
implemented behavior.

## Current behaviour (what we are changing)

- `lib/ui-server.js`:
  - `docsDir = path.join(__dirname, '..', 'docs')` (~line 198).
  - `serveDocsFile(req, res)` (~lines 442–476): strips `/docs/`, resolves under `base = path.join(docsDir, 'scuttlebot.io')`, serves the static file. For HTML it rewrites root-relative links: `html.replace(/(href|src)="\/(?!\/)/g, '$1="/docs/')` (~line 465).
  - `handleRequest` routes `req.url === '/docs' || req.url.startsWith('/docs/')` to `serveDocsFile` (~lines 488–491).
- `ssb-markdown` is already a dependency, but it is specialized for SSB posts. It rejects
  ordinary relative documentation links such as `architecture.md` and `docs/api.md`,
  rendering them as `<a class="bad">` without an `href`. It is therefore not suitable for
  the canonical docs renderer.
- None of the current canonical source docs use Markdown tables, so GFM-table fidelity is not
  a concern.

> Note: all line numbers above are indicative; confirm against the file before editing.

## Part 1 — Render `docs/*.md` and repoint `/docs`

### 1a. Audit canonical docs before publishing

Review every page in the canonical allowlist against the current code and routes before
repointing `/docs`.

- Remove stale statements that the archived scuttlebot.io site is served at `/docs`.
- Remove "to be expanded" and similar planning language from published pages.
- Verify ports, commands, plugin names, and supported behavior against the repository.
- Remove `docs/http-replication.md` from the current-docs set and from the README architecture
  links. It is a work order for an unimplemented transport, not current documentation. Keep it
  as an internal proposal unless and until that transport lands.
- Update cross-links between canonical pages so they work both on GitHub and through the docs
  renderer.

### 1b. Markdown → HTML helper

Add `markdown-it` as a direct dependency and use it for repository documentation. Keep
`ssb-markdown` for SSB post rendering.

- `renderMarkdownPage(title, mdSource)`:
  1. Render with a configured `markdown-it` instance.
  2. Rewrite intra-doc links so they resolve under `/docs/`: links of the form
     `<name>.md`, `docs/<name>.md`, and either form with `#anchor` become
     `/docs/<name>` with the anchor preserved. External links are left untouched.
     Implement this through the renderer's link rule, not a regex over rendered HTML.
  3. Reject or neutralize raw HTML in Markdown; the canonical docs do not require it.
  4. Wrap `body` in a minimal HTML template (below) and return the string.
- Page template: a single inlined-CSS document — no external assets, no build step. Requirements:
  - `<!doctype html>`, `<meta charset>`, `<meta name="viewport">`, `<title>{title} — ssbc docs</title>`.
  - A readable column: `max-width: 46rem; margin: 2rem auto; padding: 0 1rem; font: 16px/1.6 system-ui, sans-serif;`.
  - Code styling: monospace, light grey background for `code`/`pre`, horizontal scroll on `pre`.
  - Respect dark mode: `@media (prefers-color-scheme: dark)` swaps to dark background/text (reuse the `#000`/`#eef3f8` pair from ssbski for consistency).
  - A top nav strip: link back to **Docs index** (`/docs`) and to the **Archive** (`/docs/archive`).
- Keep the template as a single constant string in `lib/ui-server.js` (matches how `SSBSKI_SPLASH_STYLE` lives there). No new files, no templating engine.

Add focused renderer tests before routing tests. At minimum, prove that relative links,
anchors, external links, fenced code blocks, and escaped raw HTML render correctly.

### 1c. Canonical docs allowlist

Define the public set explicitly — these are the only `docs/*.md` exposed at `/docs`:

```
const DOC_PAGES = [
  { slug: 'overview',         title: 'Overview' },
  { slug: 'architecture',     title: 'Architecture' },
  { slug: 'api',              title: 'API' },
  { slug: 'api-reference',    title: 'API reference (generated)' }, // from Part 2
  { slug: 'cli',              title: 'CLI' },
  { slug: 'frontend',         title: 'Frontend' },

```

Everything else in `docs/` (work orders, proposals, `current-docs-proposal.md`, `scuttlebot.io-source.md`, this file) is **not** served — `/docs/<unknown>` returns 404. This prevents internal planning docs from being published.

### 1d. New routing in `serveDocsFile`

Replace the single static-serve with this dispatch (still inside `serveDocsFile`, keep the `GET`/`HEAD` guard and `..` rejection):

| Request | Response |
|---|---|
| `/docs` or `/docs/` | **Docs index**: render an intro + a `<ul>` of links to each `DOC_PAGES` slug, plus a labelled link to `/docs/archive`. |
| `/docs/<slug>` where slug ∈ allowlist | Read `docs/<slug>.md`, `renderMarkdownPage(title, src)`, serve `text/html`. |
| `/docs/archive` or `/docs/archive/...` | Serve the scuttlebot.io site exactly as today, but from `base = path.join(docsDir, 'scuttlebot.io')` with the stripped prefix `^/docs/archive/?` and the link rewrite changed to `'$1="/docs/archive/'`. Add a one-line "Archived historical reference" banner injected into each archive HTML page (one extra `html.replace` after the existing link rewrite). |
| anything else under `/docs/` | 404 via `respondNotFound`. |

The index page must make the archive's status explicit, e.g.: *"Looking for the original scuttlebot.io manual? It is preserved as a historical archive at /docs/archive and may not match this server's behaviour."*

## Part 2 — Generated, complete built-in API reference

The goal: every static manifest method mounted by the repository's built-in server stack
appears in the docs by construction, and a test fails if the docs fall behind the code.

This does **not** claim to enumerate user plugins loaded dynamically by `ssb-plugins`, because
that surface varies by installation. The generated reference must say so explicitly.

### 2a. Create one shared built-in plugin registry

Do not mirror the plugin list in a generator and then regex-parse `bin.js` to check it. Extract
the built-in `.use(...)` entries from `bin.js` into a shared module such as
`lib/builtin-plugins.js`. Both server startup and the API generator consume that registry.

The registry must preserve mount order and distinguish:
- RPC-bearing modules whose manifests belong in the generated reference,
- zero-manifest infrastructure/UI modules that still belong in server startup,
- stub/compatibility modules whose methods must be listed but clearly marked as stubs.

`lib/db` remains the root manifest loaded by `index.js`; include it explicitly in generation.
Dynamic user plugins loaded by `ssb-plugins.loadUserPlugins(...)` remain outside the committed
reference and are documented as installation-specific.

**Tradeoff:** sharing the registry makes API drift structurally difficult, but it changes the
server bootstrap path as part of documentation work. Preserve the exact existing mount order
and add a startup smoke test that confirms the server still produces its manifest successfully.

### 2b. Enumerate the built-in static surface from manifests

Every mounted module may export a static `manifest` keyed `method: type`. Current built-in
RPC-bearing groups include:

- root `lib/db`: 26 methods
- `plugins`: 5
- `gossip`: 12
- `replicate`: 2 stub methods
- `ebt`: 4
- `friends`: 9
- `blobs`: 13
- `invite`: 3
- `git`: 1
- `query`: 3
- `links2`: 2
- `ooo`: 3 stub methods

At the time this order was reviewed, that totals **83 built-in static manifest methods**.
Treat the number as an audited baseline, not a hard-coded invariant; the generator derives the
actual count from the shared registry. Namespacing: core methods are top-level (`whoami`);
plugin methods are `<name>.<method>` using the module's `name` field (`gossip.connect`,
`friends.hops`, `blobs.want`, `git.create`, `query.read`).

The shared registry is the authoritative built-in mount list. A plugin added to the registry
automatically participates in both startup and reference generation.

Caveats to encode, not paper over:
- `ssb-private1` exposes **no** RPC manifest methods; it adds box/unbox + a private-recipients path to `publish`. Document `private` as a capability of `publish`, not a `private.*` namespace.
- `query.read`, OOO, and `ssb-replicate-stub` are **stubs / no-ops** in this repo. Audit
  `links2.read` before assigning its status rather than assuming it is a stub. The reference
  must annotate non-live methods with a **status** so "complete coverage" does not
  re-introduce the "looks real but isn't" trap.
- A few callable methods come from secret-stack core rather than a static plugin manifest
  (for example `getAddress`). They are outside the generated reference's completeness claim.
  Add a short hand-written section in `docs/api.md` naming these non-manifest transport/core
  methods and explaining why they are not generated.

### 2c. Prose sidecar (incremental, hand-written)

Create `docs/api-notes.json`, a map from fully-qualified method to its documentation:

```jsonc
{
  "createFeedStream": {
    "status": "live",                       // live | stub | deprecated
    "summary": "Stream messages ordered by claimed timestamp.",
    "args": "opts: { gt, gte, lt, lte, limit, reverse, live, keys, values }",
    "example": "ssb-server createFeedStream --live --gt 0"
  },
  "query.read": {
    "status": "stub",
    "summary": "Legacy ssb-query surface. Returns an empty stream in this repo; use messagesByType / links / createHistoryStream."
  }
}
```

Only the prose is hand-authored here; the *list* of methods is never hand-maintained.

### 2d. Generator script

`scripts/gen-api-reference.js`:
1. `require` core (`../lib/db`) and consume the RPC-bearing entries from the shared built-in
   plugin registry; read each `.manifest`.
2. Build the fully-qualified method list with `type`, grouped by namespace, preserving mount order.
3. Load `docs/api-notes.json`.
4. Emit `docs/api-reference.md` with:
   - A header: `<!-- GENERATED by scripts/gen-api-reference.js — do not edit. Add prose in docs/api-notes.json and regenerate. -->`.
   - One `##` section per namespace; under it one `###` per method showing `name` and `type` (e.g. `createFeedStream — source`).
   - For each method: a **status badge** (`live` / `stub` / `deprecated`) and, if present in `api-notes.json`, the summary/args/example; otherwise the literal `_Not yet documented._`.
   - A footer counting total methods and how many have prose (a visible coverage metric).
5. Add an npm script: `"gen:api-reference": "node scripts/gen-api-reference.js"`.

`docs/api-reference.md` is committed (so `/docs/api-reference` works without a build) and listed in `DOC_PAGES`. `docs/api.md` stays as the curated narrative overview and links to the generated reference for the exhaustive list.

### 2e. Drift-guard test

`test/api-reference-current.js` (plain `node` test, matching the existing `test/*.js` style run by `npm test`):
1. Regenerate the reference in memory and assert it byte-matches the committed `docs/api-reference.md`. Failure message: "API reference is stale — run `npm run gen:api-reference`."
2. Assert the shared registry's RPC-bearing entries all appear in the generated reference,
   including stub modules and `git.create`.
3. Assert `bin.js` mounts through the shared registry rather than maintaining a second built-in
   `.use(require(...))` chain.

This is what makes "cover the complete built-in static API surface" true and *keeps* it true.

## Part 3 — Backfill prose (ongoing, low-stakes)

Not a blocker for shipping Parts 1–2. Fill `docs/api-notes.json` over time, prioritised by traffic:
1. The 9 core methods missing from `api.md` today.
2. The plugin methods used by Decent/ssbski and the CLI (`gossip.*`, `friends.*`, `blobs.want/add`, `invite.create/accept`).
3. Everything else as it is touched.

The coverage metric in the generated footer tracks progress; there is no "done" date — completeness of the *list* is guaranteed by Part 2, completeness of *prose* grows incrementally.

## Testing

- **Unit:** `test/api-reference-current.js` (2e) passes under `npm test`.
- **Startup smoke:** starting the server through the shared registry still produces the
  expected manifest and preserves plugin mount order.
- **Renderer unit checks:** relative links, anchored relative links, external links, fenced
  code, and raw HTML handling behave as specified.
- **Render checks:** for each `DOC_PAGES` slug, `GET /docs/<slug>` returns 200 `text/html` with the page title and rendered headings; intra-doc `.md` links resolve to `/docs/<slug>`; `GET /docs/<unknown>` returns 404.
- **Archive:** `GET /docs/archive` and a deep asset (e.g. `/docs/archive/css/...`) still load; archive pages show the "historical reference" banner; their internal links stay within `/docs/archive/`.
- **Visual (Playwright):** screenshot `/docs` (index), `/docs/api-reference`, and `/docs/archive` in light and dark mode; confirm readability and that dark mode applies. Save under `docs/img/` if useful for the README.
- Server-side only — no frontend rebuild. Restart the sbot to load `lib/ui-server.js` changes.

## Non-goals

- Rewriting or correcting the vendored scuttlebot.io content. It stays an archive.
- A Markdown build step or static-site generator. A small direct Markdown-rendering dependency
  is intentional because `ssb-markdown` rejects ordinary documentation links.
- Documenting third-party libraries the archive covered (pull-stream, ssb-keys, muxrpc, ssb-ref, secret-stack, community modules). Out of scope — those are not our surface.
- Syntax highlighting of code blocks server-side (the `lang-*` class is emitted; styling it is optional polish).
- Auto-generating prose. The generator only guarantees the method *list*; humans write the descriptions.
